- [ ] Confirm that v0.1.0 contains only material already approved for public v0.1 disclosure.

## Prohibited release contents

Confirm that the release contains none of the following:

- [ ] private source books or source documents;
- [ ] RAW texts, private passages or annotations;
- [ ] private corpus databases or work-level inventories;
- [ ] generated outputs derived from uncleared sources;
- [ ] substantial copyrighted quotations without an approved release basis;
- [ ] unpublished specifications;
- [ ] trademark filing evidence, receipts or correspondence;
- [ ] credentials, private configuration or API keys;
- [ ] personal, transaction or private-path information;
- [ ] the withdrawn paper, corpus-status document or incomplete flagship example.

Every box in this section must remain unchecked because the listed material must be absent.

- [ ] Protect the default branch and require review for release changes.
- [ ] Require the agreed contribution checks.
- [ ] Freeze the approved release commit.
- [ ] Generate manifests and checksums from that commit.
- [ ] Inspect the complete tag archive.
- [ ] Create an annotated or signed `v0.1.0` tag.
- [ ] Prepare the GitHub release as a draft.
- [ ] Verify release title, notes, version, date and assets.
- [ ] Enable immutable-release protection where available.
- [ ] Obtain explicit publication authorization.
- [ ] Publish the GitHub release.

## Zenodo process

- [ ] Create a manual Zenodo publication draft.
- [ ] Select a publication/technical-note resource type.
- [ ] Use William Proudler as creator unless another accepted contributor has consented.
- [ ] Set version to `0.1.0`.
- [ ] Use CC BY-SA 4.0 for the documentation deposit.
- [ ] Reserve the DOI while the record remains a draft.
- [ ] Insert the reserved DOI into the approved citation metadata.
- [ ] Upload only the checksum-matched approved assets.

- [ ] Obtain explicit publication authorization.

## Final verification

- [ ] GitHub and Zenodo titles match.
- [ ] Creator and contributor lists match.
- [ ] Version and release dates match.
- [ ] DOI and related identifiers resolve correctly.
- [ ] Uploaded file hashes match the approved manifest.

No release should be published while any required approval or verification remains incomplete.
